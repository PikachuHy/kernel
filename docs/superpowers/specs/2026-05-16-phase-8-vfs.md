# Phase 8: IPC-Native Virtual File System — Design Spec

## Overview

A virtual filesystem layer where filesystem servers are ring-3 processes communicating
via the existing IPC subsystem (Channel + Port). The kernel provides only the mount
namespace and a single `sys_open()` syscall for path-to-channel routing. All file I/O
(read, write, seek, stat, close, readdir) is direct Channel IPC between the client
and the filesystem server.

**Architecture:** Pure IPC filesystem servers — VFS namespace in kernel, everything
else is Channel messages. Files are Channels.

## Architecture

```
┌─────────────┐     ┌─────────────┐
│  init (R3)  │     │  shell (R3) │
└──────┬──────┘     └──────┬──────┘
       │ open("/foo")      │ read(h, ...)
       ▼                   ▼
┌──────────────────────────────────┐
│  KERNEL: Mount Namespace         │
│  /dev → devfs channel            │
│  /    → tmpfs channel            │
│  sys_open() routes path → FS     │
└────┬────────────────────────┬────┘
     │ Channel                │ Channel
     ▼                        ▼
┌──────────┐           ┌──────────┐
│  devfs   │           │  tmpfs   │
│  server  │           │  server  │
│  (R3)    │           │  (R3)    │
└──────────┘           └──────────┘
```

### What lives in the kernel (new code)

- **Mount namespace**: a table of `(prefix, fs_channel_handle)` entries
- **`sys_open(path, flags)`**: longest-prefix match → send Open message to FS server → return file Channel handle
- **`sys_mount(path, fs_channel_handle)`**: register/unregister mount entry

### What lives in ring 3 (new processes)

- **devfs server**: device nodes (`/dev/null`, `/dev/zero`, `/dev/console`)
- **tmpfs server**: in-memory writable filesystem, VMO-backed files and directories

### What is reused (no new kernel objects)

A file **is** a Channel (`KernelObject::Type::Channel`). No new KernelObject type
is needed. `channel_write`/`channel_read` syscalls handle all file I/O. Closing the
file's Channel handle is how you close the file.

## Mount Namespace

```cpp
struct MountEntry {
    char     path[256];        // e.g., "/dev", "/"
    handle_t fs_channel;       // Channel to the FS server for this mount
    bool     active;
};
```

The mount table is a static kernel array (small — a handful of entries for this phase).
`sys_mount(path, fs_channel_handle)` adds or replaces an entry. The kernel retains a
reference to the FS server Channel for future `sys_open` routing.

## File I/O Protocol

All file operations use a standardized message format over the file's Channel.

### Request message (client → FS server)

```cpp
struct FileMsg {
    enum Op : uint32_t {
        Open    = 0,
        Read    = 1,
        Write   = 2,
        Seek    = 3,
        Stat    = 4,
        Close   = 5,
        Readdir = 6,
    };
    Op       op;
    uint32_t flags;      // open flags or 0
    uint64_t offset;     // seek offset, readdir cookie
    uint64_t length;     // bytes to read/write
    // Channel data payload: write data, directory name for Open
};
```

### Response message (FS server → client)

```cpp
struct FileResponse {
    int32_t  result;     // 0 = success, negative = errno
    uint64_t size;       // bytes transferred, or file size for stat
    // Channel data payload: read data, stat info, dirent array
};
```

### Per-operation semantics

| Op | Request payload | Response payload | Handle transfer |
|----|----------------|-----------------|-----------------|
| Open | path string (in data) | none | Kernel transfers server-end of file Channel to FS |
| Read | none | bytes read (in data) | none |
| Write | bytes to write (in data) | none | none |
| Seek | none | none | none (response.size = new offset) |
| Stat | none | `Stat` struct (in data) | none |
| Close | none | none | Client closes handle after |
| Readdir | none | `Dirent[]` array (in data) | none |

```cpp
struct Stat {
    uint64_t size;
    uint32_t type;   // 0 = file, 1 = directory
    uint32_t mode;   // permission bits
};

struct Dirent {
    char     name[256];
    uint32_t type;   // 0 = file, 1 = directory
    uint64_t size;
};
```

### Close is implicit

When the client closes the file's Channel handle, the FS server's peer end of that
Channel is orphaned. The server detects this via a failed read/write on its end and
cleans up per-file state.

## sys_open

```
sys_open(path, flags) → handle_t (or INVALID_HANDLE on error)
```

1. Find the longest `MountEntry` whose `path` is a prefix of the requested path
2. Strip the prefix: mount at `/dev`, open `/dev/console` → remaining = `console`
3. Create a new Channel pair — call the ends `F_client` and `F_server`
4. Send an Open message over the FS server's mount Channel: `channel_write` with
   `FileMsg{Open, path=remaining, flags}` plus `F_server` as a transferred handle
5. The FS server receives the message on its end of the mount Channel, now holds
   `F_server`, and responds with a `FileResponse` ack on that same Channel
6. Read the ack on the kernel's mount Channel end — if success, return `F_client`
   to the calling process

The kernel does **not** participate in subsequent read/write on that handle — those
go directly over the Channel to the FS server.

## sys_mount

```
sys_mount(path, fs_channel_handle) → 0 or negative errno
```

1. Validate that `fs_channel_handle` is a Channel and caller has required rights
2. Add or replace entry in the mount table
3. The kernel retains a reference to this Channel for future `sys_open` routing

## Filesystem Servers

### devfs

Device nodes backed by kernel facilities. No persistent storage.

| Path | Read | Write |
|------|------|-------|
| `/dev/null` | returns 0 bytes | succeeds, discards data |
| `/dev/zero` | returns requested bytes of `\0` | succeeds, discards data |
| `/dev/console` | not implemented | writes to klog/framebuffer |

### tmpfs

In-memory writable filesystem. Files and directories are VMO-backed pages allocated
via `sys_vmo_create`. Open files track a position cursor in the server's per-file
state.

- `Open("/foo", O_CREAT)` — creates a new VMO if file doesn't exist; returns file Channel
- `Read`/`Write` operate on the VMO at the given offset, advancing position
- `Seek` updates the file position cursor
- `Readdir` returns directory entries from an in-memory list
- Directories are a special type: their VMO stores a `Dirent[]` array

### Server lifecycle

Each FS server is compiled as a separate ELF binary, embedded in the kernel image
via `llvm-objcopy` (same mechanism as `init.elf`). The kernel loads them before init.

A server's main loop:
```
while (true) {
    channel_read(server_chan) → request
    switch (request.op) {
        case Open:    create file Channel pair, send handle back
        case Read:    read from file's VMO, send data back
        case Write:   write to file's VMO, send ack
        case Seek:    update file cursor, send new position
        case Stat:    send file metadata
        case Readdir: send directory entry list
        case Close:   close file Channel, cleanup
    }
}
```

## Syscall numbering

New syscalls use numbers 50-51, continuing the existing scheme:

```
SYSCALL_OPEN   = 50   // sys_open(path, flags) → handle_t
SYSCALL_MOUNT  = 51   // sys_mount(path, fs_channel_handle) → 0/-errno
```

Existing syscalls require no changes — `channel_read`/`channel_write` serve all
file I/O.

## Boot sequence

```
Phase 5: Scheduler init
Phase 8: VFS
  1. Create Channel pair for devfs
  2. Load devfs server ELF → Process, pass server-end Channel as handle 0
  3. Mount kernel-end Channel at "/dev" via sys_mount
  4. Create Channel pair for tmpfs
  5. Load tmpfs server ELF → Process, pass server-end Channel as handle 0
  6. Mount kernel-end Channel at "/" via sys_mount
Phase 7: Load init.elf
```

init's first action: `handle_t f = sys_open("/dev/console", O_WRONLY);`

## Out of scope

- Block device layer (Phase 9+)
- Disk-backed filesystems (ext2, FAT32)
- File descriptor table (user-space libc concern)
- Permission/ownership (uid/gid)
- File locking
- mmap (future VMO map from file handle)
- Multiprocess FS serving (single server per filesystem type for now)
